# ソフトウェア工学

## 第1回
- ガイダンスを行った．
  1. ソフトウェア工学は実務から体系化されている．
  2. 知見と実践の乖離が存在する．
  3. 個人が持っているだけでは不十分でチームで共有しなければならない．

## 第2回
- ソフトウェアとは
  1. 実行されることによって必要な特性，機能，性能を提供する命令語群(コンピュータプログラム)
  2. プログラムが適切に情報を扱うことを可能とするデータ構造
  3. プログラム操作や使用法を記述した情報 

- ソフトウェアの特徴
  1. 新しい環境や技術ニーズに適応しなけれならない
  2. 新しいビジネス要求のため強化されなけれならない
  3. 近代的なシステムやDBとの相互運用のため拡張されなけれならない
  4. 進化する環境で実行できるよう、再構築されなけれならない

- ソフトウェア工学とは(IEEE)  
  ソフトウェア開発，運用，メンテナンスに対するシステマティックで規律(discipline)ある，定量化できるアプローチ適用，すなわちソフトウェアに対するエンジニアリング適用」

- ソフトウェア工学のゴール  
  「変化する状況に対応する方法論を考え出すこと．新しいソフトウェアは古いシステムから構築され，すべて互換性を持ち，他システムと相互運用されなければならない」

- プラクティスの本質
- Hookerの7原則

- ソフトウェア工学の必要性
  1. 大規模さ，複雑さ
  2. IT人材の不足
  3. 社会性と求められるミッション
  4. 不確実性の増大

- コストを考える必要がある

## 第3回
- ソフトウェアライフサイクル  
  ソフトウェアの誕生，開発・運用，廃止がある．
  1. 要件定義  
  実現する機能・実現しない機能を明確にする．
  2. 設計  
  要件定義書を設計書に変換する．
  3. 制作  
  設計書に従ってシステム構築する．  
  オフショア開発  
  賃金安い(安かった)海外企業に外部委託する手法．
  4. テストとデバッグ  
  バグないソフトウェアない．どの項目をテストするのか取捨選択が重要．
  5. 運用保守
  開発者がプロジェクトから離れていることなどを考慮する．
- ドキュメンテーション  
  文書でコミュニケーションすることが重要．

## 第4回
- プロジェクト  
  必ず終わりがあり，独自の目的を達成する仕事
  フォアキャスティング，バックキャスティング
  プロジェクトはバックキャスティングなので目標を明確に定める必要がある．
- ソフトウェア分析
  1. バグ発生率
  2. FP法
- 開発プロセス  
  一長一短であるのでよく吟味して選択する必要がある．
  1. ウォーターフォール型開発プロセス
  2. スパイラルモデル
  3. 反復型開発プロセス
  4. アジャイルプロセス  
  
## 第5，6回
- WBSとは  
  プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したもの．
- WBSの作り方
  1. スコープを明確にする
  2. 大きな作業をグルーピングする
  3. グループングした作業相互関連を考える
  4. 各グループ作業を洗い出す

## 第12回
- CI/CD
  1. CI/CDとは
    - Continuous Integration (CI)
      - コード変更を共有リポジトリに頻繁に統合するプロセス
      - 自動テストとビルドを定期的に実行し、バグ早期発見と修正を可能に
      - CI自動化により開発スムーズな進行を促進
    - Continuous Delivery (CD)
      - コード変更をテスト環境や本番環境に自動的にデプロイするプロセス
      - 自動デプロイメントを組み込み、手動操作によるデプロイ必要性を排除
      - CDにより、ユーザーフィードバックを迅に反映可能
    - CI/CDパイプライン基本的なステップ
      1. ソース: コード変更をトリガーにワークフローを起動
      2. ビルド: ソースコードをコンパイル
      3. テスト: 自動テストを実行
      4. デプロイ: テスト済みコードを本番環境にデプロイ
      5. 検証: デプロイされたアプリケーション動作確認
      6. モニタリング: 本番環境で継続的な監視
  2. Github Actionsを利用するとGithub上でもCI/CDが可能
